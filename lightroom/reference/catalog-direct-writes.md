# Editing the Lightroom catalog directly (SQLite) + dedup execution

The `.lrcat` is plain SQLite. PhotoSweeper and our metadata-rescue work edit it directly. Direct writes bypass Lightroom's processing, so **some things display and some don't** ‚Äî know which.

## Safety rules (ALWAYS)
- **Lightroom MUST be fully quit** before any write ‚Äî it locks the catalog / uses WAL, and an open LR will overwrite your edits. Verify: `pgrep -il lightroom`.
- **Back up first**, consistent copy (handles the WAL): `sqlite3 "$LRCAT" ".backup '/Users/doug/photo_tools/LightroomCatalog.bak-<ts>.lrcat'"`.
- After writing: `con.commit()` then `PRAGMA wal_checkpoint(TRUNCATE)`. Verify counts before/after.
- Queries are read-only and safe with LR open: `sqlite3.connect("file:...lrcat?mode=ro", uri=True)`.

## What WORKS via direct write
| Thing | Table.column | Notes |
|---|---|---|
| Color label | `Adobe_images.colorLabels` = `'Green'`/`'Purple'`‚Ä¶ | Displays + filterable (Attribute ‚Üí that swatch). Unlabeled = `''`. **Best tool for marking a subset to review in LR.** |
| Flag | `Adobe_images.pick` | `-1` Rejected ¬∑ `0` none ¬∑ `1` Pick. Displays. |
| Rating | `Adobe_images.rating` | 0‚Äì5; stored as float (`4.0`) ‚Äî use `int(float(x))` from CSV. |
| GPS | `AgHarvestedExifMetadata`: `hasGPS=1, gpsLatitude=<signed dec>, gpsLongitude=<signed dec>, gpsSequence=1.0` | ‚öÝÔ∏è **`gpsSequence=1.0` is REQUIRED** or the Map ignores the point. **Caveat:** even correct, a direct write shows on the **Map** but leaves the **Metadata-panel GPS *text* field blank** (the panel reads a display cache the DB write doesn't refresh). The geotag is real ‚Äî Map + export use lat/lon. For the panel field too, use the file route. |

## What you must NOT do
- **Never `DELETE FROM Adobe_images`** to remove photos ‚Äî LR's foreign keys are off, so you'd orphan rows across `AgLibraryFile`, exif, collections, keywords, stacks ‚Üí corruption. **Remove photos only via Lightroom** (it cascades correctly).
- Titles/captions are cache-sensitive ‚Äî prefer LR or the file route. **Keywords, collections, and capture-times, however, DO write cleanly at scale** ‚Äî see the next section (proven 2026-06: 25k keywords + a 314-collection set + 15k capture-time edits, no corruption).

## Creating keywords, collections & capture-times directly (proven at scale, 2026-06)
**Global id allocator.** Every new row needs a unique `id_local` from `Adobe_variablesTable` row **`Adobe_entityIDCounter`**: read it once, hand out `counter+1, +2, ‚Ä¶` for all new keywords/collections/memberships, write the final value back before commit.

**Genealogy = length-prefixed id path** (keywords AND collections). `seg(n) = str(len(str(n))) + str(n)`. Keyword root tag id is `Adobe_variablesTable['AgLibraryKeyword_rootTagID']` (16345 here). Top-level node genealogy = `/seg(root)/seg(id)` (keywords) or `/seg(id)` (collections); a child = `parentGenealogy + '/' + seg(id)` (e.g. keyword id 353658 under root 16345 ‚Üí `/516345/6353658`).

**Keywords** ‚Äî `AgLibraryKeyword`: needs `id_global` (UUID, `str(uuid.uuid4()).upper()`), `dateCreated`+`lastApplied` (Cocoa epoch = `unix ‚àí 978307200`), `genealogy`, `lc_name` (lowercased), `name`, `parent`, `includeOnExport/Parents/Synonyms=1`, `keywordType=NULL`, `imageCountCache` (count or `-1`). **`UNIQUE(parent, lc_name)`** ‚Äî dedupe case-variant names under one parent (`Las Palmas`/`las Palmas`) or the insert fails. Apply via `AgLibraryKeywordImage(id_local, image, tag)`. Aux tables (`‚Ä¶KeywordPopularity/Cooccurrence/Synonym/Face`) are caches LR rebuilds ‚Äî skip. **Tip:** parent every batch under one review keyword (`Places`, `FolderHint`) so the whole set is cull-able/reversible in one place.

**Collections & sets** ‚Äî `AgLibraryCollection`: collection `creationId='com.adobe.ag.library.collection'`, set/group `='com.adobe.ag.library.group'`; `parent=NULL` top-level else parent id; `systemOnly=0.0`; `imageCount`=member count. **`genealogy` AND `imageCount` are NOT NULL** ‚Äî passing `None` errors `NOT NULL constraint failed: AgLibraryCollection.genealogy`. A collection's genealogy is the **full length-prefixed path** = its parent set's genealogy + `'/' + seg(id)` (e.g. a month collection under a year set under `By Date`: `/75628114/75643245/75643246`), NOT just `/seg(id)` (that's only for a parent=NULL top-level collection). **A group holds mixed children** (sub-groups + collections) ‚Üí nested `By Date ‚Üí year groups ‚Üí month collections` works. Membership: `AgLibraryCollectionImage(id_local, collection, image, pick=0, positionInCollection)` ‚Äî `positionInCollection` a sortable string (`"%06d"%i`) for custom order. **To move a collection's photos into a folder on disk, do it in LR** (select-all ‚Üí drag to Folders panel); a catalog write moves nothing on disk. **To make a "photos in ‚â•2 collections" set, build a regular collection of them** ‚Äî smart collections can't express cross-collection membership (their rules only see metadata).

**Capture time / Edit Capture Time** ‚Äî authoritative is `Adobe_images.captureTime` (`YYYY-MM-DDTHH:MM:SS`), but LR's **date filter/browse reads `AgHarvestedExifMetadata.{dateYear,dateMonth,dateDay}`** (floats, month 1‚Äì12) ‚Äî **UPDATE BOTH** or the date index goes stale (every image has exactly one harvested row). `originalCaptureTime` keeps the pre-edit value ‚Äî leave it. **Trick:** stamp an unmistakable **sentinel** (`2021-06-15T03:04:00`) on a set to mark + cluster it out of the real month buckets; dump originals to CSV first for reversibility.

**Folder-name & keyword ‚Üí captureTime inference (the high-yield date-recovery move).** For untrustworthy/`dateTBD` photos, the *folder path* is the richest date clue ‚Äî mine every path segment (`folder_date_all.py`): `YYYY-MM`/`YYYY_MM`/`Month YYYY`/`YYYY Month` ‚Üí that month; a bare-year segment + a `Month`/`MM` child segment ‚Üí combine; a holiday word + year ‚Üí that holiday's month (compute **Easter** per-year via the Anonymous Gregorian algorithm; Canadian Thanksgiving=Oct); **year-only folders ‚Üí mid-year (`YYYY-06-15`)** (Doug's Aperture "year groupings" were his own year-guesses, so middle-of-year is the right default). Also scan keywords for 2-digit-year event tags (`XMAS97`‚Üí1997-12). Set `captureTime`=`YYYY-MM-15T12:00:00` + harvested `dateYear/Month/Day`, drop from `dateTBD`. One pass here dated **8,222** photos (folder ymd 2,723 + holiday 17 + mid-year 5,446 + keyword 36), cutting `dateTBD` 14,381‚Üí6,159. What's left after this is genuinely undateable ‚Äî **scanned prints** (`*Scanned` keyword, ~4,900), no-keyword loose files, and people/event-tagged-but-no-year ‚Äî needs family knowledge, not automation. **Watch the `\b` gap:** `sept4` (Sept-4) won't match `\bsept\b`; mid-year catches it anyway.

**`loc` location keywords + `gps:approx`.** Manual location-tagging vocab built as a 4-level drill-down tree `loc ‚Üí country ‚Üí region(state/prov) ‚Üí city ‚Üí place` (geocode each gazetteer place via Nominatim, reuse shared region/city nodes by `(parent_id, lc_name)` cache; `apply_loc_geo.py`). Then `apply_loc_gps.py` writes **approximate GPS** to un-geotagged `loc`-tagged photos: pick each photo's **deepest** `loc` keyword that maps to a gazetteer place ‚Üí write its coords (`gpsSequence=1.0`) + a `gps:approx` marker keyword, **never overwriting a real GPS** (`hasGPS=1` already) and **never feeding `gps:approx` back into the timeline** (output-only). Region-only tags (a province with no city) are too coarse ‚Üí skipped.

**Untrustworthy-date detection (catalog-only).** A `captureTime` is likely the scan/import/file date ‚Äî not capture ‚Äî when **no camera-model EXIF AND on a single-day mega-spike** (>250 photos that day = batch stamp: scans dated the scan day, imports the file-mod day), OR the basename is a scan series (`image####`), OR time is exactly `00:00:00`. Separates batch stamps from real busy days (a 1,162-photo Canon-7D shoot is real; 5,699 no-camera scans dated 2010-09-16 are not). ~15k of 67k flagged here.

## File route (full fidelity, when panel/keyword display must update)
Write to the photo's XMP, then in LR **Metadata ‚Üí Read Metadata from File**:
- **GPS via exiftool: use SIGNED decimals**, NOT separate Ref tags (the `-GPSLatitudeRef`/`-GPSLongitudeRef` tags are silently ignored when writing XMP ‚Üí wrong hemisphere). Correct: `exiftool -GPSLatitude=17.104 -GPSLongitude=-25.066 -o file.xmp src` ‚Üí `25,..W` ‚úì.
- Raw ‚Üí `.xmp` sidecar; JPEG/DNG/TIFF embed (rewriting a JPEG over SMB risks corruption on a mount drop).
- Read-Metadata **overwrites** catalog fields from the file ‚Äî to avoid clobbering catalog edits, **Save Metadata to File first**, add tags, then Read.

## Dedup execution gotchas (PhotoSweeper ‚Üí Lightroom)
- **Stacks hide rejected photos.** `Photo ‚Üí Delete Rejected Photos` acts on *visible* rejected ‚Äî photos buried as non-top members of **collapsed stacks** are missed (a run here: 953 hidden, ~447 left after the first delete). The grid's "X of Y photos" count being below the catalog total = collapsed stacks. **Fix:** `Photo ‚Üí Stacking ‚Üí Expand All Stacks` first, then Delete Rejected; or accept stragglers and re-run. Verify by the post-delete catalog total. (Stacks here came from the Aperture/iPhoto migration.)
- **Remove ‚âÝ Delete-from-Disk:** always **Remove** (catalog only); delete disk files separately on the NAS (deferred Phase-3 bulk pass). **Capture reject paths *before* removing.**
- **Over-grouping review:** the marked set can contain a different photo wrongly grouped. Flag = rejects with **no same-capture-time keeper**. ‚öÝÔ∏è Noisy here ‚Äî capture times are unreliable (bogus 2022/2026 import stamps), so most flagged are time-mismatched *true* dups, not real over-grouping. Narrow by **also no same-*name* keeper**. The catalog can't perceptually tell a burst frame from a dup ‚Äî **color-label the suspects and review by eye in LR.**
- **Metadata rescue reject‚Üíkeeper:** PhotoSweeper's groups aren't extractable (see `photosweeper-dedup.md`), so reconstruct pairs from the catalog by `(captureTime, baseName)`. The higher-res keeper PhotoSweeper keeps often lacks GPS its JPEG twin had ‚Üí transfer it.
- **Multi-run:** delete/empty the `Trash (PhotoSweeper)` collection before re-running so PhotoSweeper re-includes everything; loosen the Matching Level to catch what a tight pass missed (review harder ‚Äî looser over-groups more).

## "Is this disk file actually missing from my library?" ‚Äî before deleting orphans
Every folder has **more files on disk than the catalog references** ‚Äî those extras are overwhelmingly **removed-dedup-reject files Lightroom couldn't delete off the NAS**, NOT missing imports. So "uncataloged on disk" ‚âÝ "a keeper to import." Before treating a disk file as a genuine gap, check it three ways against the **live** catalog (a file pulled aside months ago looks "unique" until you re-check):
1. **Name in catalog?** (lowercased basename vs `AgLibraryFile.baseName||'.'||extension`) ‚Äî but recycled `IMG_####` names false-match, so name alone isn't proof.
2. **Byte-identical to a catalog file?** size-prefilter (only hash catalog files whose size matches a candidate) then md5 ‚Äî the gold standard for "exact copy already in library."
3. **Was it a removed reject?** match the basename against the captured `reject_*.csv` path lists (`/Users/doug/photo_tools/`) ‚Äî a hit means it *was* imported then culled as a dup (the keeper's in the library).
In one cleanup of 809 "importable, no-name-twin" leftovers: **13 byte-dups + 652 removed-rejects = 665 already represented**, only 144 genuine gaps. Caveat: byte-hash UNDER-catches non-identical representations (a lower-res JPEG whose higher-res CR2 was kept) ‚Äî when in doubt, import and let PhotoSweeper content-dedup, don't hand-judge. Move "verified redundant" to a reversible pile, never hard-`rm`, until a final purge.
